Step 3: Docker Agent, Credentials & Artifacts — Detailed Explanation
=====================================================================


What Changed from Step 2
========================

In Step 2, the entire pipeline ran inside a single python:3.11-slim container. That
worked for linting and testing, but Step 3 adds two new stages: Build Docker Image and
Push Docker Image. These stages need to run the docker command — which doesn't exist
inside the Python container.

The solution: split the pipeline into two execution contexts:

  - Python stages (checkout, install, lint, test) run inside a python:3.11-slim container
  - Docker stages (build, push) run on the Jenkins controller, which has Docker CLI
    installed (from the Dockerfile.jenkins we set up in Step 2)

This required restructuring the Jenkinsfile from a single global agent to per-stage agents.


Files Created
=============

1. app/Dockerfile
-----------------

    FROM python:3.11-slim
    WORKDIR /app
    COPY requirements.txt .
    RUN pip install --no-cache-dir -r requirements.txt
    COPY . .
    EXPOSE 5000
    CMD ["python", "app.py"]

This is the Docker image for YOUR Flask app — not the Jenkins build environment. It's what
gets built and pushed to Docker Hub so you (or anyone) can run your app with:

    docker run -p 5000:5000 rajsambasivan/flask-api:1

Line by line:
- FROM python:3.11-slim — start from the same Python base image
- WORKDIR /app — set the working directory inside the container
- COPY requirements.txt . — copy just the requirements file first
- RUN pip install --no-cache-dir -r requirements.txt — install dependencies. This is done
  before copying the rest of the code so Docker can cache this layer. If your code changes
  but requirements.txt doesn't, Docker skips the pip install on rebuild (layer caching).
- COPY . . — copy the rest of the app code
- EXPOSE 5000 — document that the app listens on port 5000 (informational, doesn't
  actually open the port)
- CMD ["python", "app.py"] — default command when the container starts


2. app/.coveragerc
------------------

    [run]
    source = app
    omit = tests/*

    [report]
    show_missing = true

Configures pytest-cov (the coverage tool):
- source = app — only measure coverage for the app module, not third-party packages
- omit = tests/* — don't count the test files themselves in coverage metrics
- show_missing = true — in the report, show which specific line numbers lack coverage


Files Modified
==============

3. Jenkinsfile
--------------

This is the big change. Let's walk through each section.


    pipeline {
        agent none
        ...
    }

In Step 2 we had agent { docker { image 'python:3.11-slim' } } at the top level, meaning
every stage ran in a Python container. Now we use agent none at the top level, which means
"don't assign a default agent — each stage will declare its own." This is needed because
different stages need different execution environments.


    environment {
        APP_NAME    = 'flask-api'
        APP_VERSION = "${env.BUILD_NUMBER}"
        REGISTRY    = 'docker.io/rajsambasivan'
    }

The environment block defines variables available to ALL stages:
- APP_NAME — the name of your Docker image
- APP_VERSION — uses Jenkins' built-in BUILD_NUMBER (auto-increments: 1, 2, 3...)
  so each build produces a uniquely tagged image (flask-api:1, flask-api:2, etc.)
- REGISTRY — your Docker Hub registry path

Together these produce image tags like: docker.io/rajsambasivan/flask-api:7


    stage('Lint & Test') {
        agent {
            docker {
                image 'python:3.11-slim'
            }
        }
        stages {
            stage('Checkout') { ... }
            stage('Install Dependencies') { ... }
            stage('Lint') { ... }
            stage('Test') { ... }
        }
    }

This is a "parent stage" that wraps four child stages. The key insight: all four child
stages share the SAME Docker container because the agent is declared on the parent. This
is why pip install in the Install stage makes flake8 and pytest available in later stages —
they're all running in the same container.

We tried putting each stage in its own docker agent first, but that failed because each
agent spins up a FRESH container — pip install in one container doesn't carry over to
the next.


    stage('Checkout') {
        steps {
            checkout scm
            stash includes: '**', name: 'source'
        }
    }

checkout scm clones the repo (same as Step 2). The new part is stash:

stash — saves a snapshot of files from the current workspace so other agents can retrieve
them later. Here we stash everything (**) under the name 'source'. This is needed because
the Docker build stage runs on a DIFFERENT agent (the Jenkins controller), which has a
different workspace directory. Without stash/unstash, the controller wouldn't have the
source code.

Think of stash like a temporary zip file stored in Jenkins' memory that any agent can
download.


    stage('Install Dependencies') {
        steps {
            sh 'pip install -r app/requirements.txt'
        }
    }

Same as Step 2 — installs flask, pytest, flake8, pytest-cov into the Python container.


    stage('Lint') {
        steps {
            sh 'flake8 app/'
        }
    }

Same as Step 2. Works because we're in the same container where pip install ran.


    stage('Test') {
        steps {
            sh 'pytest app/tests/ -v --junitxml=results.xml --cov=app --cov-report=html:coverage-report'
            stash includes: 'results.xml,coverage-report/**', name: 'test-results'
        }
    }

Two new flags compared to Step 2:
- --cov=app — enables coverage measurement for the app module
- --cov-report=html:coverage-report — generates an HTML coverage report in a
  coverage-report/ directory

The stash saves the test results and coverage report so the post block (which runs on
a different agent) can publish them.


    stage('Build Docker Image') {
        agent any
        steps {
            unstash 'source'
            sh "docker build -t ${REGISTRY}/${APP_NAME}:${APP_VERSION} ./app"
        }
    }

agent any — runs on the Jenkins controller (the only available "any" node), which has
Docker CLI installed.

unstash 'source' — retrieves the source code snapshot that was stashed during Checkout.

docker build — builds a Docker image from app/Dockerfile and tags it. For build #7, the
tag would be: docker.io/rajsambasivan/flask-api:7

Note the double quotes around the sh string — this is Groovy string interpolation, which
replaces ${REGISTRY}, ${APP_NAME}, and ${APP_VERSION} with their values BEFORE passing
the command to the shell. Single quotes would pass the variables literally (no
substitution).


    stage('Push Docker Image') {
        agent any
        steps {
            withCredentials([
                usernamePassword(
                    credentialsId: 'dockerhub-creds',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )
            ]) {
                sh '''
                    echo "$DOCKER_PASS" | docker login -u "$DOCKER_USER" --password-stdin
                    docker push ${REGISTRY}/${APP_NAME}:${APP_VERSION}
                '''
            }
        }
    }

This is the credentials stage — the core new concept in Step 3.

withCredentials — retrieves the credential you stored in Jenkins (ID: dockerhub-creds)
and injects it as environment variables (DOCKER_USER and DOCKER_PASS) ONLY within this
block. Outside the block, these variables don't exist.

Security features:
- The password is MASKED in console output — Jenkins replaces it with ****
- The variables only exist for the duration of the withCredentials block
- The actual secret is stored encrypted in Jenkins, not in the Jenkinsfile
- Anyone reading the Jenkinsfile sees 'dockerhub-creds' (the ID), never the actual token

The sh block:
- docker login — authenticates to Docker Hub using the injected credentials
  --password-stdin reads the password from stdin (piped via echo) instead of the command
  line, which is more secure (command line args are visible in process listings)
- docker push — uploads the built image to Docker Hub

Note the triple single quotes ('''): this is a Groovy multi-line string WITHOUT
interpolation. The $DOCKER_USER and $DOCKER_PASS are resolved by the SHELL (bash), not
Groovy. But ${REGISTRY}, ${APP_NAME}, ${APP_VERSION} are Jenkins environment variables,
which the shell also resolves. If we used double quotes, Groovy would try to interpolate
$DOCKER_PASS and fail because it's not a Groovy variable.


    post {
        always {
            node('') {
                unstash 'test-results'
                junit 'results.xml'
                archiveArtifacts artifacts: 'coverage-report/**', fingerprint: true
            }
        }
        success {
            echo 'All stages passed!'
        }
        failure {
            echo 'Pipeline failed — check the stage that went red.'
        }
    }

The post block changed from Step 2 in two ways:

node('') — because we used agent none at the top level, there's no agent available for
the post block. node('') explicitly allocates the Jenkins controller to run these steps.

unstash 'test-results' — retrieves the JUnit XML and coverage report that were stashed
during the Test stage (which ran in a different agent/container).

archiveArtifacts — NEW in Step 3. This saves the coverage HTML report as a build artifact
in Jenkins. Artifacts:
- Persist after the build finishes (unlike workspace files which get cleaned up)
- Are downloadable from the Jenkins build page
- fingerprint: true tells Jenkins to compute checksums so it can track which build
  produced which artifact


Key Concepts Introduced in Step 3
==================================

1. Per-Stage Agents
   Different stages can run in different environments. Python stages run in a container,
   Docker stages run on the controller. This is a common pattern: use lightweight
   containers for language-specific work, use the controller (or a dedicated agent) for
   infrastructure tasks like Docker builds.

2. stash/unstash
   Transfers files between agents. Think of it as Jenkins' internal file sharing mechanism.
   stash saves files from one agent's workspace, unstash retrieves them in another agent's
   workspace. Limited to small/medium files — for large files, use external storage or
   archiveArtifacts.

3. Credentials Management
   Secrets are stored encrypted in Jenkins and injected at runtime via withCredentials.
   They NEVER appear in the Jenkinsfile, console output, or build logs. This is how
   production CI/CD systems handle secrets — you store them once in Jenkins (or a vault),
   and pipelines reference them by ID.

4. Environment Variables
   The environment block defines variables globally. Jenkins also provides built-in
   variables like BUILD_NUMBER, JOB_NAME, WORKSPACE, etc. Using BUILD_NUMBER as the
   image tag ensures every build produces a unique, traceable Docker image.

5. Build Artifacts
   archiveArtifacts saves files beyond the build's lifetime. Unlike the workspace (which
   is temporary), artifacts are accessible from the Jenkins UI forever (until manually
   deleted). Common artifacts: test reports, coverage reports, compiled binaries, Docker
   image manifests.

6. Docker Image Layer Caching
   In app/Dockerfile, we COPY requirements.txt before COPY . . This is intentional.
   Docker builds images in layers, and each instruction creates a layer. If a layer's
   inputs haven't changed, Docker reuses the cached layer. By copying requirements.txt
   first and running pip install, Docker can skip the slow pip install step when only
   your code changes (but requirements.txt stays the same).


How It All Fits Together (mapped to Build #7 console output)
==============================================================

Each numbered step below corresponds to what you saw in the console output of build #7.


1. You click Build Now
   Console: "Started by user admin"
   (line 1)


2. Jenkins reads the Jenkinsfile from your repo
   Console: "Obtained Jenkinsfile from git https://github.com/graycoldknight/jenkins-learning.git"
   (line 2)
   Jenkins clones the repo FIRST just to read the Jenkinsfile. This is a lightweight
   fetch — it only needs the pipeline definition to know what to do next.


3. Jenkins checks if the python:3.11-slim image exists locally
   Console: "docker inspect -f . python:3.11-slim"
   (line 35)
   docker inspect checks if the image is already cached on the host. The "." output means
   it exists — no need to pull it again. If it wasn't cached, you'd see a docker pull here.


4. Jenkins starts a Python container for the "Lint & Test" parent stage
   Console: "docker run -t -d -u 0:0 -w /var/jenkins_home/workspace/flask-api-pipeline
             --volumes-from 185763198f5f... python:3.11-slim cat"
   (line 41)
   Breaking down the docker run flags:
   - -t -d — run in detached mode with a terminal (container stays alive in background)
   - -u 0:0 — run as root (UID 0, GID 0) inside the container
   - -w /var/jenkins_home/workspace/flask-api-pipeline — set the working directory
   - --volumes-from 185763198f5f... — mount ALL volumes from the Jenkins container.
     This is how the Python container sees the workspace — it shares the Jenkins
     container's filesystem via volume mounts. The long hex string (185763198f5f...) is
     the Jenkins container's ID.
   - python:3.11-slim cat — start the container running "cat" (which just keeps it alive
     waiting for input). Jenkins will exec commands into this running container for each
     stage.

   The container ID c8dbca34dc1f... is now running and waiting for commands.


5. Checkout stage — clone the repo inside the Python container
   Console: "git fetch ... https://github.com/graycoldknight/jenkins-learning.git"
             "Checking out Revision 6b63fb3..."
             "Stashed 41 file(s)"
   (lines 46-65)
   Jenkins runs git commands to checkout commit 6b63fb3 (the "nest Python stages" commit).
   Then stash saves all 41 files to Jenkins' internal storage for later use by the Docker
   build stage.

   Note the warning on line 50: "special launcher ... will be ignored" — this means git
   runs on the Jenkins controller, not inside the Python container. Git operations use the
   controller's git binary because the Docker workflow plugin can't reliably run git inside
   the container. This is fine — the files end up in the shared workspace either way.


6. Install Dependencies — pip install inside the Python container
   Console: "+ pip install -r app/requirements.txt"
             "Collecting flask==3.1.0..."
             "Successfully installed Jinja2-3.1.6 ... flask-3.1.0 ... pytest-8.3.5 ..."
   (lines 70-137)
   This runs INSIDE the Python container (c8dbca34dc1f). pip downloads and installs all
   packages. The "+" prefix before each command is sh -x trace output — Jenkins runs
   shell commands with tracing enabled so you can see exactly what executed.

   The "WARNING: Running pip as root" is harmless — this container is disposable, so
   there's no risk of breaking system packages.


7. Lint stage — flake8 inside the same Python container
   Console: "+ flake8 app/"
   (line 143)
   flake8 runs and produces no output — meaning zero lint violations. If there were
   violations, you'd see them listed here and the stage would fail. This works because
   pip installed flake8 in step 6, and we're in the SAME container.


8. Test stage — pytest with coverage inside the same Python container
   Console: "+ pytest app/tests/ -v --junitxml=results.xml --cov=app --cov-report=html:coverage-report"
             "app/tests/test_app.py::test_health PASSED    [ 33%]"
             "app/tests/test_app.py::test_greet PASSED      [ 66%]"
             "app/tests/test_app.py::test_add PASSED        [100%]"
             "Coverage HTML written to dir coverage-report"
             "3 passed in 0.33s"
             "Stashed 12 file(s)"
   (lines 148-168)
   pytest runs all 3 tests, writes JUnit XML to results.xml, and generates an HTML
   coverage report in coverage-report/. Then stash saves 12 files (results.xml + the
   HTML report files) under the name 'test-results' for the post block.


9. Python container is destroyed
   Console: "docker stop --time=1 c8dbca34dc1f..."
             "docker rm -f --volumes c8dbca34dc1f..."
   (lines 172-173)
   Jenkins stops and removes the Python container. It served its purpose — lint and tests
   are done. The stashed files survive because they're stored in Jenkins, not in the
   container.


10. Build Docker Image stage — runs on the Jenkins controller
    Console: "Running on Jenkins in /var/jenkins_home/workspace/flask-api-pipeline"
              "+ docker build -t docker.io/rajsambasivan/flask-api:7 ./app"
    (lines 183-184, 207)
    No "withDockerContainer" here — this runs directly on the Jenkins controller (which
    has Docker CLI from our Dockerfile.jenkins). The unstash retrieves the source code,
    then docker build creates the app image.

    The build output shows Docker executing each Dockerfile instruction:
    - #5 [1/5] FROM python:3.11-slim — uses cached base image (DONE 0.6s)
    - #6 [2/5] WORKDIR /app — creates /app directory (DONE 0.1s)
    - #7 [3/5] COPY requirements.txt — copies requirements file (DONE 0.2s)
    - #8 [4/5] RUN pip install — installs dependencies (DONE 17.3s — the slow step)
    - #9 [5/5] COPY . . — copies app code (DONE 0.3s)
    - #10 naming to docker.io/rajsambasivan/flask-api:7 — tags the final image

    Note: pip install happens TWICE in this pipeline — once in the Python container for
    testing (step 6), and once in the Docker build for the deployable image (step 10).
    These are two different contexts with two different purposes.


11. Push Docker Image stage — docker login and push on the Jenkins controller
    Console: "Masking supported pattern matches of $DOCKER_PASS"
              "+ echo ****"
              "+ docker login -u rajsambasivan --password-stdin"
              "Login Succeeded"
              "+ docker push docker.io/rajsambasivan/flask-api:7"
    (lines 352-428)

    First, notice line 352: "Masking supported pattern matches of $DOCKER_PASS" — Jenkins
    is telling you it will replace any occurrence of your Docker Hub token with **** in the
    output. This is withCredentials in action.

    Line 355: "+ echo ****" — your actual token was piped to docker login, but Jenkins
    masked it. The real command was: echo "<your-token>" | docker login -u rajsambasivan
    --password-stdin

    "Login Succeeded" — authenticated to Docker Hub.

    Then docker push uploads the image layers:
    - "Mounted from library/python" (lines 418-422) — Docker Hub recognizes that some
      layers (the base python:3.11-slim layers) already exist in Docker Hub's storage,
      so it doesn't upload them again. It just "mounts" (references) them. This saves
      significant upload time and bandwidth.
    - "Pushed" (lines 423-427) — these are YOUR layers (requirements install, app code)
      that are unique to your image and need to be uploaded.
    - "7: digest: sha256:34c4213a..." — the final image manifest is stored. The digest
      is a content-addressable hash — anyone pulling this exact image gets this exact hash.


12. Post block — publish results on the Jenkins controller
    Console: "Running on Jenkins in /var/jenkins_home/workspace/flask-api-pipeline"
              "[Pipeline] unstash"
              "[Pipeline] junit"
              "Recording test results"
              "[Pipeline] archiveArtifacts"
              "Archiving artifacts"
              "Recording fingerprints"
              "All stages passed!"
    (lines 439-452)

    node('') allocates the Jenkins controller. unstash retrieves the test results that
    were saved back in step 8. junit parses results.xml and creates the Test Result tab
    in the Jenkins UI. archiveArtifacts saves the coverage report as downloadable build
    artifacts. fingerprints are checksums Jenkins uses to track artifact lineage.

    Finally: "All stages passed!" — the success post condition fires.

    "Finished: SUCCESS" (line 458) — build #7 is complete.
